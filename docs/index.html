Integration tests you can read

htest runs browser integration tests defined as YAML manifests — ansible/drill style. It builds a run graph from task prerequisites, runs in dependency order, and makes reruns safe with idempotent tasks.

Get started → Follow the tutorial

Two mantras drive the design: speed (eager page loads, batched DOM queries, implicit waits instead of fixed sleeps) and repeatability (idempotency gates, deterministic ordering, sessions always closed cleanly).

# A whole test, top to bottom
tasks:
  - name: homepage_loads
    steps:
      - goto: "https://example.com"
      - wait_for: "h1"
      - assert: { selector: "h1", exists: true }
      - screenshot: "home.png"

What's in the box

  • YAML manifests — steps, prerequisites, idempotency, variables.
  • Run graph — topological order from needs; cycles rejected up front.
  • Two backends — a deterministic mock for developing manifests, and a real WebDriver backend (Firefox/Chrome).
  • Hierarchical selectors — text matching and scoped descent that plain CSS can't express.
  • Implicit waits — no flaky fixed sleeps; poll until the condition holds.
  • Loops — loop: repeats a task over a range or list, expanded into real tasks at load time.
  • Multi-file + cross-file deps — split suites across files, depend across them.
  • Playbooks — one CI file collects suites, sets shared settings, and can own the driver.
